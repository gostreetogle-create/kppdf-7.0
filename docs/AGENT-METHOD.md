# AGENT-METHOD.md — Метод прохождения проекта

> **Назначение.** Пошаговый алгоритм, по которому ИИ-агент проходит по проекту: от первой встречи до сдачи артефакта. Два режима: «Быстрый старт» (5+5+5 мин) и «Полное погружение» (~30 мин).
>
> **Обязательное условие:** до ознакомления агент ОБЯЗАН прочитать `00_START_HERE.md` и `AGENT-ROLES.md`.

---

## 0. Контекст

Метод прохождения строится на 4 принципах:

1. **Не дублируй сделанное** — проверь `PROJECT-STATE-LOG.md` + уже существующие файлы **ДО** начала работы.
2. **Не ломай чужую роль** — границы `AGENT-ROLES.md` §2 соблюдаются свято.
3. **Фиксируй дыры** — каждая найденная дыра = запись в `PROJECT-STATE-LOG.md` (глобальная) или `00-spr/00-otkrytye-voprosy.md` (локальная).
4. **Спроси PO вместо угадывания** — если есть любое multi-choice решение, спроси через `ask_user`, не выбирай сам.

---

## 1. Два режима прохождения

### 1.1 «Быстрый старт» — 15 минут

Используй, если задача **узкая** (один файл, один статус, одно правило) и **ты уже знаком с проектом**.

| # | Шаг | Время | Что делать |
|---|---|---|---|
| 1 | Прочитай `00_START_HERE.md` + `PROJECT-STATE-LOG.md` (последние 5 записей) | 5 мин | Контекст «куда ветер дует» |
| 2 | Прочитай `AGENT-ROLES.md` §2 (роль) | 5 мин | Проверь что ты в своей роли |
| 3 | Сделай задачу + пройди `AGENT-REVIEW.md` | 5 мин | Самопроверка перед сдачей |

### 1.2 «Полное погружение» — ~30 минут

Используй, если задача **большая** (новый модуль, новый документ, новый статус) или **ты впервые в проекте**.

| # | Шаг | Время | Что делать |
|---|---|---|---|
| 1 | Прочитай `00_START_HERE.md` + `CHECKLIST.md` (весь) | 10 мин | Общая структура и маршруты |
| 2 | Прочитай `PROJECT-STATE-LOG.md` (последние 10 записей) | 5 мин | Audit trail |
| 3 | Прочитай `AGENT-ROLES.md` полностью + свою роль в `AGENT-PROMPTS.md` | 5 мин | Роль + шаблон промпта |
| 4 | Прочитай `AGENT-FORMAT.md` §1 (П1–П8) и §5 (A1–A11) | 5 мин | Стиль + запреты |
| 5 | Сделай задачу + пройди `AGENT-REVIEW.md` | 5 мин | Самопроверка |
| 6 | Напиши запись в `PROJECT-STATE-LOG.md` (если изменение глобальное) | 2 мин | Audit |

## 2. Пошаговый алгоритм прохождения задачи

### 2.1 Старт

**⛔ НЕ НАЧИНАЙ РАБОТУ**, пока не выполнены 3 проверки:

1. **Проверка «уже сделано?»** — прочитай `PROJECT-STATE-LOG.md` последние 10 записей. Может, твоя задача уже решена предыдущим агентом.
2. **Проверка «не дублирую»** — глобальный поиск по проекту (grep) по ключевым словам задачи. Если нашёл — открой и проверь, актуально ли решение.
3. **Проверка «в своей роли»** — сопоставь задачу с `AGENT-ROLES.md` §2. Если задача межролевая → декомпозируй или верни её в `PROJECT-STATE-LOG.md` со ссылкой.

### 2.2 Исполнение

Работай в своей роли. **Не трогай файлы чужих ролей.** Если увидел, что соседний файл надо править — создай запись в `PROJECT-STATE-LOG.md` или `00-spr/00-otkrytye-voprosy.md`, не правь сам.

### 2.3 Дробление больших файлов (Правило 3.1)

> **Правило.** Если создаваемый или редактируемый файл вырастает > 400 строк — **остановись и разбей на 2–3 файла с явными cross-link**.

Симптомы:
- файл > 400 строк,
- больше 6–7 разделов верхнего уровня,
- 2 разные темы в одном файле.

Что делать:
1. Определи естественные границы тем.
2. Создай новые файлы (или попроси Архитектора через `PROJECT-STATE-LOG.md`).
3. Свяжи их cross-link.
4. Старый файл → помечай как DEPRECATED или удаляй через git.

### 2.4 Завершение (Pre-action / Post-action checklist)

**Pre-action чек-лист** (ПЕРЕД началом изменения):
- [ ] Я прочитал `00_START_HERE.md` + `PROJECT-STATE-LOG.md` (последние 10)?
- [ ] Я проверил, что задача не дублирует уже сделанное?
- [ ] Я в своей роли (см. `AGENT-ROLES.md` §2)?
- [ ] Целевой файл / директория существует (или я создаю через STUB)?
- [ ] Я знаю формат целевого файла (см. `AGENT-FORMAT.md`)?

**Post-action чек-лист** (ПЕРЕД сдачей):
- [ ] Файл проходит `AGENT-REVIEW.md` §1–§5?
- [ ] Размер ≤ 400 строк (hard limit)?
- [ ] Каждая таблица имеет шапку?
- [ ] Каждое поле имеет тип + источник?
- [ ] Нет «вероятно», «может быть», «обычно» (анти-паттерн A2)?
- [ ] Нет вводных абзацев > 3 строк без конкретики (A3)?
- [ ] Все кросс-ссылки указывают на существующие файлы/разделы?
- [ ] Записал PSL-NNN (если изменение глобальное)?

## 3. Обработка противоречий (Правило 4.1)

> **Правило 4.1.** Если находишь противоречие между двумя документами или между документом и кодом — НЕ молчи, НЕ правь молча.

Алгоритм:

1. Зафиксируй противоречие в `PROJECT-STATE-LOG.md` с типом `terminology` / `critical_fix` / `schema`.
2. Укажи: **Файл A** vs **Файл B**, **что именно расходится**, **следствие** (что сломается).
3. Спроси PO через `ask_user` (или список в multi-choice).
4. Зафиксируй решение PO как новый PSL-NNN с reference на исходный.

Если противоречие **критическое** (сломает prod/codegen) — пометь `critical_fix` и заблокируй дальнейшую работу до решения PO.

## 4. Фиксация дыр в покрытии (Правило 5.1)

> **Правило 5.1.** Каждая найденная дыра = запись в одном из двух мест:
> - **Проектная дыра** (затрагивает > 1 модуль или общую схему) → `PROJECT-STATE-LOG.md`
> - **Локальная дыра** (внутри одного модуля) → `<модуль>/00-spr/00-otkrytye-voprosy.md`

### 4.1 Шаблон OQ (Open Question)

```markdown
### OQ-NNN — <краткая суть> [YYYY-MM-DD]

| Поле | Значение |
|---|---|
| ID | OQ-NNN |
| Дата | YYYY-MM-DD |
| Модуль | <модуль или "Универсально"> |
| Автор | <роль + ник> |
| Приоритет | 🔴 P0 / 🟡 P1 / 🟢 P2 |
| Связанные PSL | PSL-XXX (если есть) |

**Контекст.** <что происходило, где нашли>

**Суть дыры.** <что не покрыто>

**Варианты решения.**
- A) ...
- B) ...
- C) ...

**Рекомендация.** <если есть>

**Статус.** 🔵 OPEN / 🟡 PENDING DECISION / 🟢 RESOLVED
```

### 4.2 Когда заводить OQ, а когда править сразу

| Ситуация | Действие |
|---|---|
| Нашёл однозначный баг (опечатка, неработающая ссылка) | Правь сразу, потом запиши в PSL тип `critical_fix` |
| Нашёл стилистическую проблему (неясная формулировка) | Правь сразу (in-place) |
| Нашёл противоречие между двумя источниками | OQ + спроси PO |
| Нашёл дыру в покрытии (правило не описано) | OQ + задай в `00-spr/00-otkrytye-voprosy.md` |
| Нашёл архитектурную дыру (> 1 модуль затронут) | PSL-NNN + спроси PO |

## 5. STUB-exception (правило для пустых файлов-заглушек)

> **Определение STUB.** Пустой файл-заглушка, созданный агентом для обозначения места в структуре. Содержит только маркер `⚠️ STUB: создан агентом <роль>, требует наполнения`.

### 5.1 Когда создавать STUB

| Ситуация | Создавать STUB? |
|---|---|
| Архитектор спроектировал дерево папок, но других ролей ещё не было | ✅ ДА, для каждого файла в дереве |
| Аналитик понял что нужно правило, но ещё не готов его сформулировать | ✅ ДА, для правила |
| Увидел что в чужом файле нужна секция, которой нет | ❌ НЕТ — это не твоя зона, создай OQ |

### 5.2 Обязательный формат STUB

```markdown
# <имя файла>

⚠️ **STUB:** создан агентом <роль> (< дата >), требует наполнения <целевой ролью>.

## 0. Контекст
<краткое описание того, что будет в файле после наполнения>

## 1. <Заголовок первой секции>
_TODO: наполнение._
```

### 5.3 Правила для STUB

1. Кто создал STUB — тот больше его **не трогает** (после того как другая роль начала наполнение).
2. STUB должен быть **≤ 30 строк** (это заглушка, не черновик).
3. В `PROJECT-STATE-LOG.md` STUB-creation фиксируется как PSL с типом `structure`.
4. Наполнение STUB = отдельный PSL с типом `process` или `schema`.

## 6. Правило автономности (5.3)

> **Правило 5.3 (автономность).** Агент может действовать автономно для **низкорисковых** изменений. Для **средне- и высокорисковых** — обязан спросить PO.

| Риск | Кто решает | Примеры |
|---|---|---|
| 🟢 Низкий | Агент сам | Опечатки, форматирование, добавление edge-case в существующий файл |
| 🟡 Средний | Спросить PO через `ask_user` | Новый статус, новый тип поля, изменение формата таблицы |
| 🔴 Высокий | Обязательно спросить PO, и только после согласия действовать | Изменение RBAC, удаление Prisma-модели, изменение стека, удаление большого файла |

**Фиксация решения PO** — запись в `PROJECT-STATE-LOG.md` со ссылкой на multi-choice (`ask_user` результат).

## 7. UX-принципы для документов (5.4)

Даже в чисто технической документации соблюдай UX для читателя:

| Принцип | Что значит |
|---|---|
| **Сначала «куда я попал»** | Первый раздел любого документа — контекст (2–3 строки: что это, для кого) |
| **Сначала «как прочитать»** | Если файл большой — дай оглавление в §0 |
| **Сначала «зачем мне это»** | Каждый файл начинается с `> **Назначение.**` |
| **Один раздел = одна идея** | Если секция описывает 2 разные идеи — разбей на 2 |
| **Примеры > абстракции** | Где возможно — дай конкретный пример рядом с правилом |
| **Связи > копии** | Если контент уже есть в другом файле — дай cross-link, не копируй |
