# AGENT-ROLES.md — Роли и pipeline агентов

> **Назначение.** Описание 7 специализированных ролей ИИ-агентов, их зон ответственности и pipeline передачи артефактов. **Универсально для любого проекта**, не привязано к конкретной бизнес-логике.
>
> **Обязательное условие:** до ознакомления с этим файлом агент ОБЯЗАН прочитать `00_START_HERE.md` (главная точка входа).

---

## 0. Контекст

Каждый агент берёт **одну роль в одной сессии**. Смена роли = новая сессия с явным заявлением. Роли **не пересекаются по ответственности** (каждая — своя зона).

Атомарность защищает от LLM-дрейфа. Параллельное выполнение ролей = риск расхождения контекста → последовательное выполнение обязательно.

---

## 1. Атомарная ответственность и исключения

> **⚠️ Атомарная ответственность.** Агент **не имеет права** менять файлы вне своей компетенции. Если видишь, что нужно изменить файл другой роли — создай тикет в `00-spr/00-otkrytye-voprosy.md` (локальный) или в `PROJECT-STATE-LOG.md` (глобальный).

> **Исключение для STUB-файлов (см. `AGENT-METHOD.md` §5).** Создание **пустого файла-заглушки** с маркером `⚠️ STUB: создан агентом, требует проверки координатором` **разрешено**, если файл ещё не существовал. Это не редактирование чужого файла — это создание новой инфраструктуры. Как только другая роль начнёт наполнять STUB, создатель больше его НЕ трогает.

> **Подробнее** об этих правилах — в [`AGENT-METHOD.md`](AGENT-METHOD.md) §5.

## 2. Роли агентов

### 2.1 Архитектор (Architect)

| | |
|---|---|
| **Цель** | Файловая иерархия, зависимости между папками, пререквизиты |
| **Вход** | Список существующих документов по теме; общая папка проекта |
| **Выход** | Обновлённое дерево папок; матрица зависимостей; нет циклов и дублирования |
| **Проверяет** | • Каждый файл имеет ВХОДНЫЕ ДАННЫЕ (откуда) и РЕЗУЛЬТАТ (что получается)<br>• Порядок чтения внутри папки не нарушает зависимостей (00 → 01 → 02 → 03 → 04)<br>• Нет двух файлов с одной темой |
| **Не делает** | Содержимое файлов, бизнес-правила, UX |

### 2.2 Бизнес-аналитик (Business Analyst)

| | |
|---|---|
| **Цель** | Бизнес-правила, state-машины, RBAC-матрицы, инварианты |
| **Вход** | ТЗ от продакт-оунера; существующая документация для контекста |
| **Выход** | Явные правила с номерами (`Правило 1.`); state-машина с диаграммой |
| **Проверяет** | • У каждого правила есть уникальное следствие при нарушении<br>• Каждый статус имеет определение, переходы и доступные действия<br>• RBAC-матрица покрывает все действия для всех ролей<br>• Нет «магических констант» без обоснования |
| **Не делает** | Техническую архитектуру, SQL-схему, дизайн экранов |

### 2.3 Моделировщик данных (Data Modeler)

| | |
|---|---|
| **Цель** | Сущности, поля с типами, связи, snapshot-логика |
| **Вход** | Бизнес-правила; список операций |
| **Выход** | Документ «Модель данных модуля»: сущности + таблица FK + ON DELETE правила |
| **Проверяет** | • Каждое поле имеет тип и описание откуда оно берётся<br>• Каждый FK имеет явное ON DELETE правило<br>• Snapshot-поля явно отмечены<br>• Уникальность там, где она нужна |
| **Не делает** | Бизнес-правила, UX, RBAC |

### 2.4 UX-проектировщик (UX Designer)

| | |
|---|---|
| **Цель** | Экраны, формы, кнопки, потоки пользователя, превью |
| **Вход** | State-машина от бизнес-аналитика; согласованная UX-концепция |
| **Выход** | Списки полей экранов с типами и источниками; ASCII-схемы потоков |
| **Проверяет** | • Каждый экран имеет список полей с типами и источниками данных<br>• Каждая кнопка имеет состояния (доступна в статусах X/Y/Z)<br>• Live preview показывает то, что будет в артефакте<br>• Никаких модальных окон без обоснования |
| **Не делает** | SQL, RBAC, общую архитектуру |

### 2.5 QA-валидатор (QA Validator)

| | |
|---|---|
| **Цель** | Edge-кейсы, инварианты, дыры в покрытии, тестовые сценарии |
| **Вход** | Готовые (или почти готовые) документы модуля |
| **Выход** | Список найденных дыр с приоритетом (🔴 P0 / 🟡 P1 / 🟢 P2); список тест-сценариев |
| **Проверяет** | • Что произойдёт, если пользователь сделает X (10 типичных сценариев)<br>• Каждое правило покрыто хотя бы 1 тест-сценарием<br>• **ОБЯЗАТЕЛЬНО:** минимум один тест-кейс, который ЛОМАЕТ текущую логику<br>• Нет неподдерживаемых комбинаций входных данных |
| **Не делает** | Написание тестов; исправление найденных дыр |

### 2.6 Технический писатель (Tech Writer)

| | |
|---|---|
| **Цель** | Ясность языка, единая терминология, форматирование |
| **Вход** | Сырые документы от других агентов |
| **Выход** | Финальные формулировки (правки **in-place в существующих файлах**, никаких новых файлов) |
| **Проверяет** | • Каждое утверждение — правило/поле/переход/таблица<br>• Термины единые (из глоссария)<br>• Каждый файл начинается с контекста и определения<br>• Нет двух файлов, описывающих одно и то же |
| **Не делает** | Содержание (только формулировка); структуру папок (Архитектор) |

### 2.7 Координатор (Coordinator)

| | |
|---|---|
| **Цель** | Интеграция результатов всех ролей + устранение дублей + кросс-модульная проверка |
| **Вход** | Результаты всех 6 ролей выше |
| **Выход** | Кросс-модульный отчёт в табличном формате (шаблон в `AGENT-PROMPTS.md` §5) |
| **Проверяет** | • Кросс-ссылки работают (файл A ссылается на поле файла B, и оно существует)<br>• Терминология единая между всеми файлами<br>• Нет противоречий между модулями |
| **Не делает** | Переписывание документов (только отчёт с конкретными исправлениями) |
| **Handoff** | После отчёта → задача Архитектору (структурные) или Аналитику (правила/инварианты) 🔴. Без явного handoff петля обрывается. |

## 3. Pipeline ролей и полный цикл

Роли запускаются **последовательно**, не параллельно:

```
[Архитектор]──▶[Бизнес-аналитик]──▶[Моделировщик]──▶[UX]──▶[Тех.писатель]──▶[QA]──▶[Координатор]
                                                                                       │
                                                                                  отчёт
                                                                                       ▼
                                                                  [Координатор]────отчёт────▶[Архитектор/Аналитик]
                                                                                                         │
                                                                                              правят по типу находок
                                                                                                         ▼
                                                                          [Архитектор/Аналитик]────исправления────▶[QA]  ◀──── ПЕТЛЯ ОБЯЗАТЕЛЬНАЯ
                                                                                                                                   │
                                                                                                                              перепроверка
                                                                                                         [Координатор]──────▶[Координатор]────отчёт v2
```

**Правило 3.1 (полный цикл).**

1. **Architect → BusinessAnalyst → DataModeler → UX → TechWriter → QA → Coordinator** — основная цепочка.
2. **Coordinator отчёт → Architect или BusinessAnalyst** (по типу находок). Структурные противоречия → Architect; правила/инварианты → BusinessAnalyst.
3. ← **После правок — ОБЯЗАТЕЛЬНАЯ петля через QA-валидатор** для перепроверки, а не отдача в Coordinator напрямую. Architect/Analyst фиксит → возвращает QA → QA подтверждает или находит остатки → Coordinator собирает отчёт v2.
4. Цикл повторяется, пока QA-валидатор не подтвердит «🔴 P0 больше нет».
5. Только после условий шага 4 артефакт считается готовым к передаче разработчику.

| # | Роль | Что производит | Что передаёт следующей |
|---|---|---|---|
| 1 | Архитектор | Структура папок, матрица зависимостей | Список файлов → Бизнес-аналитику |
| 2 | Бизнес-аналитик | Пронумерованные правила, state-машина | Правила → Моделировщику |
| 3 | Моделировщик | Сущности, поля с типами, FK | Дамп сущностей → UX-проектировщику |
| 4 | UX-проектировщик | Поля экранов + состояния | Список экранов → Тех.писателю |
| 5 | Тех.писатель | Финальные формулировки, единая терминология (in-place) | Чистые файлы → QA-валидатору |
| 6 | QA-валидатор | Список дыр + тест-сценарии (включая минимум 1 ломающий) | Отчёт по дырам → Координатору |
| 7 | Координатор | Кросс-модульный отчёт (см. `AGENT-PROMPTS.md` §5) | Если есть 🔴 P0 → Architect или BusinessAnalyst |

**Цикл завершается** когда QA-валидатор подтверждает «новых 🔴 P0 дыр нет».
