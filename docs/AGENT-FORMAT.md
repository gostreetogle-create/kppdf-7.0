# AGENT-FORMAT.md — Принципы, форматирование и анти-паттерны

> **Назначение.** Правила того, **как** должны быть написаны все документы проекта: 8 принципов, правила форматирования заголовков / таблиц / ссылок и список запрещённых действий.
>
> **Обязательное условие:** до ознакомления с этим файлом агент ОБЯЗАН прочитать `00_START_HERE.md`.

---

## 0. Контекст

Этот файл объединяет три темы с общим корнем — «как писать текст правильно»:
- **Принципы** (П1–П8) — общие максимы.
- **Правила форматирования** — конкретный формат.
- **Анти-паттерны** (A1–A11) — чего НЕ делать.

Все эти правила — обязательны для ЛЮБОГО документа в проекте.

## 1. Принципы (П1–П8)

| # | Принцип | Что это значит |
|---|---|---|
| П1 | **Точность > нарратив** | Лучше 3 строгих правила, чем 10 абзацев пояснений |
| П2 | **Цифры, а не слова** | «Ставка НДС 20%», а не «примерно пятая часть» |
| П3 | **Явные зависимости** | «Файл A зависит от файла B, потому что C» — в начале каждого документа |
| П4 | **Самодостаточность** | Каждый документ можно прочитать отдельно и понять без чтения всего проекта (но обязательно со ссылками) |
| П5 | **Кросс-ссылки вместо копий** | «См. `01_*/00-spr/00-orgs.md` поле Наценка по умолчанию», а не дублировать определение |
| П6 | **Нумерация правил** | Каждое правило имеет номер (`Правило 1.`, `Правило 2.`) — для ссылок из тестов и кода |
| П7 | **Поля с типами** | Никаких «имя» без типа. Всегда: `имя: String (обязательное)` или `price: Decimal ≥ 0` |
| П8 | **Сценарии = шаги** | Любой сценарий — это нумерованный список шагов с входами / выходами |

### 1.1 Язык

- **Русский** — основной язык документа (пользователи — русскоязычные).
- **Термины** — канонические из локального глоссария модуля (когда появится `99_Spravochniki/GLOSSARY.md`).
- **Технические термины** (FK, ON DELETE, API endpoint, RBAC) — на английском, без перевода.

### 1.2 Что считается «водой» (запрещено)

- ❌ «В прошлой версии это бесило менеджеров» → конкретное правило.
- ❌ «Это эталон, который мы воспроизводим один-в-один» → описание поведения.
- ❌ «Здесь происходит магия» → механизм (что подставляется, откуда берётся).
- ❌ Длинные абзацы введения без конкретики → список правил.
- ❌ «Возможно», «вероятно», «может быть» → либо точно, либо НЕ описывать.
- ❌ Исторические экскурсы (откуда пришло решение) → в `PROJECT-STATE-LOG.md` (audit trail) или в `OPEN-QUESTIONS-LEGACY.md` (DEPRECATED архив).

## 2. Заголовки

### 2.1 Структура

```markdown
# Название (одно предложение — что это)

> **Назначение.** Что это и зачем.

## 0. Контекст / Глоссарий
## 1. Первая тема
### 1.1 Подтема
### 1.2 Подтема
## 2. Вторая тема
...
```

**Запрещено:** бездумные `####` без содержания. Сам `####` **допустим** для атомарных полей-меток, но всегда с обоснованием.

### 2.2 Шаблон раздела (стандартный)

Любой новый документ начинается так (жирным — то, что меняется):

```
# <Название раздела>

> **Назначение.** Что это и зачем.

## 0. Контекст
[1–2 абзаца: что такое, откуда данные, что на выходе]

## 1. Определение / Сущность
[Чёткое определение, границы, что входит / не входит]

## 2. Поля / Атрибуты
| Поле | Тип | Обязательное | Описание | Источник |
| ... | ... | ... | ... | ... |

## 3. Правила
**Правило 1.** [Условие] → [Следствие]. [Пример если нужно]

## 4. Переходы (если применимо)
| Откуда | Куда | Кнопка / Действие | Условие | Кто может |
| ... | ... | ... | ... | ... |

## 5. Edge-кейсы
| Сценарий | Что должно произойти |
| ... | ... |

## 6. Связанные документы
- Ссылка 1
```

## 3. Правила таблиц

- **Каждая таблица имеет шапку** (даже если она очевидная).
- **Колонки в логическом порядке:** сначала «что это», потом «тип», потом «правила», потом «пример».
- **Нет пустых ячеек** — пиши `—` или `не применимо`.
- **Сортировка:** статусы по порядку lifecycle (а не алфавиту).

---

## 4. Правила ссылок

- Внутренние ссылки Markdown — относительные: `[SCHEMA.md](99_Spravochniki/SCHEMA.md)`.
- Ссылка на конкретный раздел: `[§3 Модель данных](99_Spravochniki/SCHEMA.md#3-модель-данных)` (если якорь работает — лучше использовать; если нет — открой файл и найди раздел вручную).
- Поле из другой таблицы: «См. `Organization.vatRate` в `99_Spravochniki/SCHEMA.md`».
- **Не используй backticks внутри текста ссылки** (`[текст](url)`) — ломает рендеринг. Внутри ссылки — plain text. Backticks **вокруг** ссылки допустимы.

## 5. Анти-паттерны (A1–A11)

| # | Анти-паттерн | Почему плохо | Что делать вместо |
|---|---|---|---|
| A1 | Писать код | В фазе документации код теряет актуальность | Только `.md`, таблицы, ASCII-схемы |
| A2 | Писать «вероятно», «обычно», «может быть» | Размывает ответственность | Точное правило или ничего |
| A3 | Длинная вводная без конкретики | Иллюзия глубины без содержания | Первый раздел = контекст (2–3 строки) |
| A4 | Исторические пассы («в v3 делали так») | Не чертёж, дневник разработки | В `PROJECT-STATE-LOG.md` как PSL-NNN или в `OPEN-QUESTIONS-LEGACY.md` |
| A5 | Дублирование в разных файлах | Дрейф при изменениях | Один файл + кросс-ссылка |
| A6 | Утверждения без обоснования | «Нужно потому что нужно» | Обоснование: правило / сценарий / ссылка на OQ |
| A7 | Таблицы без шапки | Неполнота данных | Каждая таблица имеет шапку |
| A8 | Модальные окна без обоснования | Избыточная прерывистость UX | Каждое модальное окно — с явной целью |
| A9 | Многостраничные файлы (> 400 строк) | Невозможно охватить целиком (hard limit) | Дробление на 2–3 файла + cross-link |
| A10 | «Пустые» разделы (заголовок без содержания) | Иллюзия покрытия | Заполнить или удалить |
| A11 | Самоцитирование («как было сказано выше», «напомним») | Трата токенов, засоряет документ | Сразу к сути. Если нужна связь — прямая ссылка |
