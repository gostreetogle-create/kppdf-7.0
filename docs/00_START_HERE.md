# 00_START_HERE.md — Точка входа для ИИ-агента

> **Назначение.** Этот документ — **главная входная точка** для любого агента (ИИ или человек), заходящего в проект `kppdf-7.0`. Содержит ответ на 3 вопроса: кто ты, как работать, когда «готово».
>
> **Когда читать:** ПЕРВЫМ, до любых других документов проекта.
>
> **Объём:** ≤ 200 строк target / 400 hard limit (см. `AGENT-REVIEW.md` §1.6).

---

## 0. Про что этот документ

1. **Кто ты** — какую роль берёшь на себя (Архитектор / Аналитик / Моделировщик / QA / Координатор / PO).
2. **Как работать** — по какому методу проходишь по проекту (шаги + STUB).
3. **Когда «готово»** — по каким критериям оцениваешь свою работу перед сдачей.

Полные ответы — в 5 топических файлах (см. §2 «Навигация»).

### 0.1 PROJECT-STATE-LOG.md — журнал инкрементальных изменений

Каждое **глобальное** изменение проекта (новый модуль, изменение стека, добавление политики, переименование папки, новый статус) **должно быть записано** в `PROJECT-STATE-LOG.md` с указанием: **Дата** / **ID (`PSL-NNN`)** / **Тип** / **Модуль** / **Описание** / **Причина** / **Затронутые файлы** / **Автор** / **Связанные OQ**.

**Правило:** каждый агент при старте читает `00_START_HERE.md` (этот файл) **и** `PROJECT-STATE-LOG.md` (последние 10 записей). Это даёт контекст «куда ветер дует» без перечитывания всей базы.

## 1. Контекст проекта

### 1.1 Что такое `kppdf-7.0`

`kppdf-7.0` — это **greenfield-проект**. Бизнес-логика намеренно НЕ унаследована от `kppdf-6.0` (только методология проведения проекта). Цель документации в `docs/` — дать любому ИИ-агенту, открывшему проект, **самодостаточный инструментарий** для:

- понимания структуры проекта без чтения всего кода,
- принятия решений в рамках ответственности своей роли,
- безопасного ведения проекта без потери контекста,
- соблюдения сквозных правил (форматирование, чек-листы, журнал изменений).

### 1.2 Главная боль (какую решаем методология)

| Было (в типичных проектах без документации для AI) | Стало (после применения `docs/`) |
|---|---|
| AI лезет в файлы не своей роли — ломает чужое | Атомарная ответственность ролей (§ см. `AGENT-ROLES.md`) |
| AI забывает что уже сделано, дублирует решения | `PROJECT-STATE-LOG.md` + проактивная проверка «уже сделано?» |
| AI пишет «воду» вместо правил | Принципы П1–П8 + анти-паттерны A1–A11 |
| Документы разрастаются, теряют актуальность | Hard limit 400 строк + обязательная декомпозиция |
| Контекст утерян, новый сеанс AI не знает с чего начать | Протокол возобновления (`CHECKLIST.md` §6) |

### 1.3 Главный принцип документации

> **Каждое утверждение в документе — это либо правило, либо поле, либо переход, либо таблица, либо сценарий «если X → то Y».**
>
> «Вода» (нарратив без конкретики, истории «что пытались в прошлом», «бесило менеджеров») — **запрещена** (анти-паттерн A3).

## 2. Навигация по инструкциям

Все правила и шаблоны для работы агентов вынесены в **5 топических файлов**. **Читай их по необходимости:**

| # | Файл | Когда читать | Содержимое |
|---|---|---|---|
| 1 | [`AGENT-ROLES.md`](AGENT-ROLES.md) | Перед началом работы | 7 ролей, атомарность + исключение STUB, pipeline + полный цикл |
| 2 | [`AGENT-METHOD.md`](AGENT-METHOD.md) | Перед началом работы | Шаги прохождения + правила фиксации дыр + шаблон OQ |
| 3 | [`AGENT-FORMAT.md`](AGENT-FORMAT.md) | При написании любого документа | Принципы П1–П8 + форматирование + анти-паттерны A1–A11 |
| 4 | [`AGENT-REVIEW.md`](AGENT-REVIEW.md) | Перед сдачей работы | Чек-лист MUST/SHOULD (включая hard limit 400 строк) |
| 5 | [`AGENT-PROMPTS.md`](AGENT-PROMPTS.md) | При запуске нового агента | Промпт-шаблоны для 5 ролей + табличный формат отчёта Координатора |

**Минимальный маршрут для типовой задачи:**
1. Этот файл (5 мин)
2. [`AGENT-ROLES.md`](AGENT-ROLES.md) (5 мин)
3. Выполнение задачи (основная работа)
4. [`AGENT-REVIEW.md`](AGENT-REVIEW.md) перед сдачей (5 мин)

**Полное погружение** — в [`CHECKLIST.md`](CHECKLIST.md) + [`AGENT-METHOD.md`](AGENT-METHOD.md) §2.1.

## 3. Структура папок проекта

### 3.1 Структура папки `/docs/` (только методология)

```
docs/
├── 00_START_HERE.md           ← ЭТОТ ФАЙЛ (читать первым)
├── AGENT-ROLES.md             ← Роли и pipeline
├── AGENT-METHOD.md            ← Метод прохождения + STUB
├── AGENT-FORMAT.md            ← Форматирование и анти-паттерны
├── AGENT-REVIEW.md            ← MUST/SHOULD чек-лист самопроверки
├── AGENT-PROMPTS.md           ← Промпт-шаблоны для запуска ролей
├── CHECKLIST.md               ← Мастер-навигатор (только маршруты)
└── PROJECT-STATE-LOG.md       ← Журнал PSL-NNN (audit trail)
```

> **Дерево всего проекта** (включая `frontend/`, корневой `README.md`, будущий `backend/`) — в [`../README.md`](../README.md). Здесь — только структура `/docs/`.

### 3.2 Как структура будет расти

> ⚠️ Это **ожидаемая** структура для будущих модулей. PO может утвердить иную номенклатуру при появлении первого модуля.

По мере появления модулей проекта структура `docs/` будет расширяться по шаблону:

```
docs/
├── (методология — 8 файлов выше, стабильны)
└── (бизнес-документация — добавляется по мере надобности)
    ├── 01_<модуль>/             ← модуль N (по аналогии с другими проектами)
    │   ├── README.md                # Точка входа модуля
    │   ├── 00-spr/                  # Справочники
    │   ├── 01-shablon/              # Шаблоны
    │   ├── 02-konstruktor/          # Сборка / UI
    │   ├── 03-zhiznennyj-cikl/      # Процесс
    │   └── 04-pravila/              # Ограничения и права
    └── 99_Spravochniki/         ← Сквозные справочники
        ├── SCHEMA.md                # Схема БД (когда появится)
        ├── RBAC-MATRIX.md           # Права (когда появится)
        └── GLOSSARY.md              # Единый словарь
```

**До появления бизнес-модулей** структура остаётся ровно такой, как в §3.1. Не плодить пустые папки.
