# CHECKLIST.md — Мастер-навигатор проекта

> **Назначение.** Единая точка входа для КАЖДОГО запуска ИИ-агента в проекте. Содержит маршруты, карту ролей и протоколы. **НЕ дублирует** содержимое `00_START_HERE.md`, `AGENT-*.md`, `PROJECT-STATE-LOG.md` — только маршрутизирует к ним.
>
> **Объём:** ≤ 250 строк (target) / 400 (hard limit per `AGENT-REVIEW.md` §1.6).

---

## 0. Самообслуживание чек-листа (правила ведения)

**Когда обновлять этот файл** — только если изменилась **глобальная маршрутизация** (новая роль, новый документ-источник, новый МОДУЛЬ). Для точечных правок — не трогать, обновите источник по ссылке.

**Когда НЕ обновлять:**
- Изменился один STUB в каком-то модуле → править сам STUB, не этот файл.
- Закрылся один OQ → править глоссарий модуля или OQ-мастер (когда появится).
- Закрылся один PSL-NNN → править `PROJECT-STATE-LOG.md`, не этот файл.

**Правила содержания:**

1. **Не дублировать** таблицы решений и журнал PSL — только ссылки (анти-паттерн A5).
2. **Каждый маршрут** = одна строка «Если хочешь X → читай Y» (анти-паттерн A3 — без вводных абзацев).
3. **Все варианты выбора** (multi-choice от AI в конце ответа) — фиксируются через `ask_user` или добавляются в `PROJECT-STATE-LOG.md` как запись PSL-NNN («Причина» = «зафиксирован выбор пользователя»).
4. **Hard limit:** при превышении 250 строк → немедленно разбить на `CHECKLIST.md` + `CHECKLIST-TRACKER.md` с явной cross-link (анти-паттерн A9).
5. **Формат:** следовать `AGENT-FORMAT.md` (П1–П8): нумерация правил, цифры вместо слов, кросс-ссылки вместо копий.

## 6. Протокол возобновления (потеря контекста)

**Если контекст утерян** (обрыв интернета, новая сессия, краш):

1. **Открой `docs/CHECKLIST.md`** (этот файл — точка входа).
2. **Прочитай `docs/PROJECT-STATE-LOG.md`** — последние 5 записей PSL-NNN сверху вниз. Определи последний завершённый шаг.
3. **Спроси PO ровно одну фразу:** *«Контекст восстановлен. Последний шаг был **PSL-NNN — <название>**. Продолжить с того же места, начать новую задачу или закрыть run?»*
4. **Если PO говорит «продолжить»:** открой соответствующий документ по маршруту §5 и следуй его инструкциям.
5. **Если нужно сменить роль:** см. §4 → новый запуск через соответствующий промпт-шаблон в `AGENT-PROMPTS.md`.
6. **Если задача уже сделана** (PSL-NNN = DONE): переходи к следующему шагу / новой задаче без повторов.

**Признак восстановления успешно:** PO подтвердил «continue» + ты видишь §3 текущего состояния + последний PSL-запись совпадает с памятью.

## 7. Решения PO (точки выбора)

**Все варианты multi-choice, которые AI предлагает в конце своего ответа, должны быть ЗАФИКСИРОВАНЫ** в одном из двух уровней:

| Уровень | Где фиксировать | Когда |
|---|---|---|
| **1. Журнал изменений** | [`PROJECT-STATE-LOG.md`](PROJECT-STATE-LOG.md) новая запись PSL-NNN | При глобальном структурном изменении, выборе стека, выборе роли |
| **2. Локальные дыры модуля** | `<модуль>/00-spr/00-otkrytye-voprosy.md` (когда появится) | При гранулярной дыре реализации в одном модуле |

**Сводный индекс ЗАКРЫТЫХ решений:** (в этом проекте пока отсутствует — будет создан `99_Spravochniki/OPEN-QUESTIONS-MASTER.md`, когда появится первый бизнес-модуль).

**Правило:** если AI предлагает выбрать A / B / C — **никогда не выбирай сам**. Спроси PO через `ask_user`. После ответа — зафиксируй решение в одном из 2 уровней выше со ссылкой `PSL-NNN`.

## 8. Архив завершённых пунктов

**Правило: НЕ удалять.** Завершённые элементы складируются в архивные зоны:

| Архивная зона | Что туда |
|---|---|
| `git rm` + git history | исходные файлы, которые распускаются на подфайлы |
| `PROJECT-STATE-LOG.md` §1 «Журнал изменений» | все PSL-NNN (только сверху вниз, никогда не редактировать снизу) |
| `archive/` подпапка в `/docs/` (если создана в будущем) | версионированные отчёты прошлых версий методологии |

**Перед архивированием:** убедись что информация зафиксирована в git через commit + push. **После архивирования:** файл у получателя не должен содержать ссылок на архивное (анти-паттерн A11 «самоцитирование» — архив служит историей, не активной навигацией).

## 9. «Задавай умные вопросы» (защита от ломания сделанного)

Прежде чем что-то менять, спроси себя (через `ask_user` если неуверен):

| Вопрос | Когда задать | Если ответ «не знаю» |
|---|---|---|
| Это сделано? | Каждый раз перед стартом (§2) | Сначала проверь, потом спрашивай PO |
| Это сломает PSL-NNN? | Перед edit / rm любого `.md` в `/docs/` | Найди PSL-NNN для файла → см. причину → спроси PO |
| Это hard limit (250 / 400 строк)? | Перед созданием `.md` > 200 строк | Разбей на 2–3 файла + cross-link |
| Это в правильной роли (Правило 3.1)? | Принимая задачу | Проверь `AGENT-ROLES.md` §2 атомарность — если пересекается с двумя ролями, разбей на sub-task |
| Какая папка для нового документа? | Перед созданием нового файла | См. `00_START_HERE.md` §3 «Как структура будет расти» |

**Правило «не ломай сделанного»:** если ответ на любой из вопросов «да» — остановись и спроси PO через `ask_user`. Если PO уверен «override» — фиксируй новое решение как новый PSL-NNN со ссылкой на переопределённую запись.
